# PRD-002a: 대시보드 확장 계획 (Future)

## 배경

PRD-002에서는 MVP 범위로 단순한 수익률·환율 처리·이벤트 모델을 채택했다. 이 문서는 **MVP 이후에 도입할 확장 기능들의 방향성과, 그 도입을 가능하게 하기 위해 PRD-002 단계에서 미리 갖춰둬야 할 데이터 구조**를 정리한다.

지금 당장 구현하지 않지만, MVP 데이터 모델이 이 확장들을 막지 않도록 사전 검토하는 것이 목적이다.

## 확장 기능 목록

### 1. 시간가중수익률 (TWR)

**문제**
PRD-002의 단순 수익률(`순수익 / 원금 × 100`)은 입출금 타이밍에 따라 왜곡된다. 작은 자금일 때 좋은 판단을 했고 큰 자금이 들어온 뒤 시장이 빠지면, 단순 수익률은 나쁘게 나오지만 실제 판단의 질은 우수했을 수 있다.

**해결**
시간가중수익률(Time-Weighted Return)은 입출금 영향을 제거해 순수 투자 판단의 결과만 측정한다. 하위 기간(sub-period)별 수익률을 기하평균으로 합성한다.

```
TWR = Π_i (1 + r_i) − 1
  r_i = (기말가치 − 입출금) / (기초가치) − 1   (각 sub-period i)
```

**MVP 사전 준비**

- [x] `daily_snapshots` 테이블에 매일 평가액·원금·예수금 기록 (PRD-002에 포함됨)
- [ ] sub-period 기준이 되는 입출금 이벤트는 이미 `account_events`로 식별 가능

이 둘이면 추후 별도 컬럼 추가 없이 소급 계산 가능.

---

### 2. 벤치마크 비교

**문제**
"내 수익률이 KOSPI나 S&P 500 대비 잘했나?"를 알 수 없다. 절대 수익률만으로는 시장 일반의 흐름과 자신의 판단을 구분할 수 없다.

**해결**
자산 히스토리 그래프에 벤치마크 지수의 정규화된 라인을 오버레이. 사용자가 첫 투자한 시점을 100으로 정규화하고, 같은 기간 동안 KOSPI / S&P 500 / 비트코인 등의 변화를 같은 축에 그림.

**MVP 사전 준비**

- [x] `prices` 테이블에 지수도 같은 스키마로 저장 가능 (`ticker = "KOSPI"`, `asset_type = "index"` 등 추가)
- [ ] `prices.asset_type` enum에 `index` 값 추가 가능성을 염두에 두고 설계

UI 측 변경만 필요하며, 데이터 모델은 거의 그대로 활용 가능.

---

### 3. 실현/미실현 손익 분리

**문제**
PRD-002의 "순수익"은 평가손익(미실현)과 매도로 확정한 손익(실현)이 섞여 있다. 세금 계산이나 심리적 구분(실현된 이익 vs 평가상 이익)에 부적합.

**해결**
KPI 카드와 그래프에서 두 값을 분리해 표시.

```
실현 손익 = Σ (매도 정산금 − 매도 시점 평균매수가 × 매도 수량) − 매도 수수료/세금
미실현 손익 = Σ (현재가 × 보유수량 − 평균매수가 × 보유수량)
순수익 = 실현 + 미실현
```

**MVP 사전 준비**

- [ ] 매도 이벤트 저장 시 `realized_pnl` 필드를 `account_events`에 미리 추가하면 성능 개선. 다만 MVP에선 거래 횟수가 적어 on-the-fly 계산도 충분함. **PRD-002에 컬럼은 추가하지 않고, 추후 ALTER TABLE로 추가하기로 함**.

### 4. 환차익 / 종목차익 분리 (관점 3)

**문제**
PRD-002는 외화 자산을 항상 현재 환율로 환산(관점 1). 따라서 평가액 변동에는 환율 변동분과 종목 가격 변동분이 섞여 있다. "내 종목 선택이 좋았나" vs "환율이 도와줬나"를 구분할 수 없다.

**해결**
세 가지 손익을 분리한다.

```
종목 손익 = Σ (현재가 − 매수가) × 수량 × 매수 시점 환율
환차익   = Σ (현재가 × 수량 × 현재 환율 − 현재가 × 수량 × 매수 시점 환율)
실현 환차익 = 매도 시 (매도 시점 환율 − 매수 시점 환율) × 매도 정산금
```

KPI 카드에 "종목 수익 / 환차익" 두 카드를 분리해 표시.

**MVP 사전 준비**

- [x] `account_events.fx_rate_at_event` 필드 (PRD-002에 포함됨) — 매수·매도 시점의 환율을 함께 기록해두면 소급 분리 계산 가능

이게 핵심이다. 이 필드 없이 시작하면 소급 분리가 불가능하므로 MVP 단계에서 반드시 기록해야 함.

### 5. 외환 (KRW ↔ USD 환전) 이벤트

**문제**
미국 주식을 사려면 KRW를 USD로 환전해야 한다. PRD-002는 입금/출금만 모델링하므로, 환전을 "KRW 출금 + USD 입금" 두 개의 이벤트로 입력해야 한다. 이 경우 원금이 한 번 빠졌다가 다시 들어오는 것으로 보여 부자연스럽고, 환율 차익(환전 스프레드 손실 등)도 추적되지 않는다.

**해결**
별도 이벤트 타입 `fx_exchange` 도입.

| 필드 | 내용 |
|---|---|
| from_currency / from_amount | 환전 전 통화·금액 |
| to_currency / to_amount | 환전 후 통화·금액 |
| spread_loss | (선택) 시중 환율과의 차이로 발생한 손실 |

효과:
- 예수금: from 통화 −, to 통화 +
- 원금: 변동 없음 (자기자본 내에서의 통화 전환이므로)
- spread_loss는 누적 수수료에 가산할지 별도 항목으로 둘지 결정 필요

**MVP 사전 준비**

- [ ] `account_events.event_type` enum에 `fx_exchange` 값 추가 가능성 염두. enum 확장은 마이그레이션으로 처리 가능하므로 차단 요소 아님.

---

### 6. 부문별 자산 비중 파이 차트

**문제**
자산 히스토리 그래프는 시계열만 보여줄 뿐, 현재 시점에서 한국주식 / 미국주식 / 코인의 비중이 어떻게 분포되어 있는지 한눈에 보이지 않는다.

**해결**
대시보드에 파이 차트 섹션 추가. 부문별 평가금액 비중을 %와 절대 금액으로 표시.

**MVP 사전 준비**
없음. 종목 카드 데이터로 즉시 계산 가능.

> MVP에서 제외한 이유: 자산 히스토리 그래프의 예수금 라인이 이미 "현금 vs 주식" 비중 정보를 시각화하므로, 부문별 비중까지 동시에 보여주면 정보 밀도가 과해질 수 있음. 사용해 본 뒤 실제로 필요해지면 추가.

---

### 7. 기업 액션 자동 감지 / 알림

**문제**
주식 분할이나 무상증자가 발생하면 사용자가 직접 `corporate_actions` 테이블에 기록해야 한다. 누락 시 보유수량이 잘못 계산됨.

**해결**
일별 종가 cron이 가격의 비정상적 점프(예: 하루에 75% 하락 + 거래량 정상)를 감지하면 기업 액션 후보로 자동 표시. 사용자에게 알림 → 사용자가 확인하여 등록.

**MVP 사전 준비**

- [x] `corporate_actions` 테이블 (PRD-002에 포함됨)
- [ ] 자동 감지 로직은 별도 구현. MVP 단계에선 사용자 수동 입력 UI만 우선.

---

### 8. 종목별 수익률 분석

종목 카드를 탭했을 때 진입하는 개별 종목 페이지는 **별도 PRD-003**(예정)에서 다룬다. 이 PRD의 범위는 대시보드 차트·KPI에 한정.

## 우선순위 (잠정)

도입 가치 / 구현 복잡도 / 사용자 학습 곡선을 고려한 잠정 순서:

1. **부문별 파이 차트** — 가장 단순, 즉시 가치
2. **벤치마크 비교** — 데이터 모델 거의 그대로, UI만 추가
3. **실현/미실현 손익 분리** — 거래 횟수 늘면 가치 증가
4. **환차익 / 종목차익 분리** — 미국 주식 비중 커지면 가치 증가
5. **외환(환전) 이벤트** — 4번과 함께 도입
6. **TWR** — 입출금 빈도 늘어난 후
7. **기업 액션 자동 감지** — 보유 종목 수 늘어난 후

각 항목의 도입 시점은 실제 사용 데이터에 기반해 재평가한다.
